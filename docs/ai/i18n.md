# Internationalization

How to translate UI strings in Open Library. Server-rendered text uses Templetor's `$_()` directly. Client-rendered text (Lit components, vanilla JS UI, anything that builds DOM in the browser) needs the bridge pattern below.

## Server-rendered strings

In a `.html` template, wrap any user-visible string in `$_()`:

```html
<h2>$_("Recently added")</h2>
<button>$_("Borrow")</button>
```

Strings are extracted to `openlibrary/i18n/messages.pot` by the `generate-pot` pre-commit hook and translated per locale under `openlibrary/i18n/<lang>/`.

## Client-rendered strings: the `data-i18n` bridge

JavaScript modules cannot use Templetor's `$_()` — and the JS-side `ugettext` helper in this codebase is **only a pass-through**, not a real translation call. Strings hardcoded in JS ship in English everywhere.

The working pattern is to render translated strings server-side into a `data-i18n` attribute on the consuming element, then read them in JS at startup. The component module also exports English defaults that double as both the source for `.pot` extraction (via the partial) and the runtime fallback.

### 1. Write a `_i18n.html` partial that emits a JSON dict

Create a small template that returns a JSON object of translated strings. By convention these live next to the feature template and are named `<feature>_i18n.html`.

```html
$def with ()
$# Translated chrome strings for the header search modal (SearchModal.js).
$# Rendered into the trigger's `data-i18n-ui` attribute and read by
$# search-modal/constants.js (searchModalStringsFromElement). The keys and
$# English source strings here must match DEFAULT_SEARCH_MODAL_STRINGS in that
$# same file.
$ search_modal_i18n = {
$     "inputPlaceholder":  _("Search books, authors…"),
$     "closeAria":         _("Close search"),
$     "noResults":         _("No results found"),
$     "removeFilter":      _("Remove filter: %s"),
$ }
$json_encode(search_modal_i18n)
```

The leading comment is important: it tells the next contributor which JS file consumes the partial and which constant the English fallback lives in. Keep keys and English text in lockstep across the two files.

### 2. Drop the JSON onto the consuming element

In the parent template, render the partial into a `data-*` attribute on the element your JS module already targets (a trigger button, a container, etc.). Use `$:render_template` (the `$:` prefix is required to skip HTML-escaping of the JSON):

```html
<button
  type="button"
  class="search-bar-trigger"
  aria-label="$_('Search')"
  data-i18n="$:render_template('search/availability_i18n')"
  data-i18n-ui="$:render_template('search/search_modal_i18n')"
></button>
```

When one element needs two independent payloads, use two attribute names (e.g., `data-i18n` for the option list, `data-i18n-ui` for chrome strings). Don't merge them — they usually correspond to two separate concerns with separate fallbacks.

### 3. Read and merge in JS, with English defaults as the fallback

Keep the canonical English strings in the JS module. They serve three purposes: source-of-truth for the partial's keys, runtime fallback if the attribute is missing or malformed, and what ships when the page is rendered in English.

```js
export const DEFAULT_SEARCH_MODAL_STRINGS = {
    inputPlaceholder: 'Search books, authors…',
    closeAria: 'Close search',
    noResults: 'No results found',
    removeFilter: 'Remove filter: %s',
};

export function searchModalStringsFromElement(el) {
    let overrides = null;
    try {
        const raw = el?.dataset?.i18nUi;
        if (raw) overrides = JSON.parse(raw);
    } catch { /* fall through to defaults */ }
    return overrides
        ? { ...DEFAULT_SEARCH_MODAL_STRINGS, ...overrides }
        : DEFAULT_SEARCH_MODAL_STRINGS;
}
```

Spreading the overrides over the defaults means a partial translation (some keys present, others missing) never blanks out a string — the English shows through for any key the locale didn't translate.

Wrap the `JSON.parse` in `try`/`catch` and treat any failure as "fall back to defaults." Don't throw — a bad attribute on one element shouldn't take down the page.

### 4. Use the merged strings to build DOM

```js
import { searchModalStringsFromElement } from './constants.js';

const trigger = document.querySelector('.search-bar-trigger');
const strings = searchModalStringsFromElement(trigger);

input.placeholder = strings.inputPlaceholder;
closeButton.setAttribute('aria-label', strings.closeAria);
```

For strings with placeholders (`%s`), interpolate client-side with `sprintf` from the project's existing helpers — don't reinvent.

## Choosing where strings live

| String source | Where to put it |
| --- | --- |
| Visible text in a `.html` template | `$_("…")` directly in the template |
| Static text in a Lit component's `static styles` content (rare) | Pass in via attribute from the template, where `$_()` is available |
| Dynamic UI strings built in JS (labels, ARIA, status messages) | `_i18n.html` partial → `data-i18n*` attribute → JS reader (this doc) |
| Counts, numbers, prices | Format client-side; don't translate |

If a Lit component is the consumer, the `data-i18n*` attributes go on the host element (the custom-element tag), and the component's reader runs in its `connectedCallback` or constructor.

## Jinja (`.jinja`) templates — preferred over Templetor

Jinja2 is the preferred template engine for new UI in Open Library. It replaces Templetor (`.html` templates), which is legacy. The Jinja environment (configured in `openlibrary/core/jinja.py`) uses the same per-request translation system as Templetor, so all existing locale files work seamlessly.

> **New work should use `.jinja` templates.** Templetor is maintained for backward compatibility only.

### How it works

The Jinja environment is configured in `get_jinja_env()` (`openlibrary/core/jinja.py`):

- `jinja2.ext.i18n` extension is enabled
- `install_gettext_callables(gettext, ngettext, newstyle=True)` wires the project's per-request `.mo` translations (via `load_translations(lang).ugettext`) into Jinja
- `ext.i18n.trimmed = True` strips leading/trailing whitespace from `{% trans %}` block content at render time
- No manual `env.globals` registration needed — `install_gettext_callables` auto-registers `_`, `gettext`, and `ngettext`

### Two translation pathways

| Syntax | What it calls | Use case |
|---|---|---|
| `{% trans %}...{% endtrans %}` | Installed gettext callable | Longer strings with HTML or multiple placeholders |
| `{{ _('…') }}` / `{{ gettext('…') }}` / `{{ ngettext('…', '…', n) }}` | Same installed callable (auto-registered) | Short inline strings, attribute values |

All four pathways go through the same callables — `{{ _() }}` is not a special override, unlike Templetor where `$_()` is a special construct.

### Guidelines

#### Use named placeholders only

Newstyle gettext only supports `%(name)s` placeholders — **no positional `%s`**. This is a feature: named placeholders carry semantic meaning for translators.

```jinja
{# GOOD #}
{{ _('Look for this edition for sale at %(store)s', store=store.name) }}

{# BAD — positional %s won't work with newstyle #}
{{ _('Look for this edition for sale at %s') | format(store.name) }}
```

#### Simple inline HTML in strings is fine

For simple inline links (a single `<a>` tag within a sentence), embed the HTML directly in the string and use `|safe`. This is the pattern used by **all other translated strings** in the codebase — ~97 strings with `<a>` tags follow this approach.

```jinja
{# GOOD — simple, matches codebase conventions #}
{{ _('Learn more <a href="/help">here</a>.') | safe }}

{# OVERKILL — variable names are longer than the HTML they replace #}
{% trans link_start='<a href="/help">'|safe, link_end='</a>'|safe %}
Learn more {{ link_start }}here{{ link_end }}.
{% endtrans %}
```

> **When to extract HTML into variables:** Only if the HTML structure is complex enough that a translator could easily break it — e.g., multiple nested tags, several attributes on different elements, or structural markup like `<table>` / `<ul>` within the string. For a single inline link, keep it simple.

> **Note:** Do NOT use `|safe` on user-supplied data — only on string literals or trusted template variables.

#### Avoid `{% trans %}` blocks for short strings with inline HTML

For short strings with a single inline HTML tag, prefer `{{ _('...') | safe }}` — it's simpler, matches the 97+ existing translated strings with `<a>` tags, and avoids the verbosity of variable extraction.

```jinja
{# GOOD — simple, matches codebase conventions #}
<small>{{ _('When you buy books using these links the Internet Archive may earn a <a class="nostyle" href="/help/faq/about#selling">small commission</a>.') | safe }}</small>

{# OVERKILL — verbose variable names, djLint false positives #}
{% trans
commission_link_start='<a class="nostyle" href="/help/faq/about#selling">'|safe,
commission_link_end='</a>'|safe -%}
When you buy books using these links the Internet Archive may earn a {{ commission_link_start }}small commission{{ commission_link_end }}.
{%- endtrans %}
```

Reserve `{% trans %}...{% endtrans %}` for longer multi-line strings that still need HTML, where the readability benefit of the block form outweighs the verbosity of variable extraction.

#### Control whitespace in `{% trans %}` blocks

`ext.i18n.trimmed` strips whitespace at render time, but **Babel's extraction does not apply this policy**. The extracted `.pot` entry must match what Jinja looks up at runtime, or translations will never be found.

Use `-%}` on the opening tag and `{%-` on the closing tag to strip the surrounding whitespace from the extracted string:

```jinja
{# GOOD — extracted string is clean, matches runtime lookup #}
{% trans -%}
This is a multi-line paragraph that needs to stay in a block form
for readability. The surrounding whitespace is trimmed by `-%}` / `{%-`.
{%- endtrans %}

{# BAD — extracted POT entry includes leading/trailing whitespace,
   translation key won't match what Jinja looks up at runtime #}
{% trans %}
    {{ _('This indented string would include newlines in the POT entry.') }}
{% endtrans %}
```

After adding or changing a Jinja template, regenerate the POT file:

```bash
docker compose run --rm home python ./scripts/i18n-messages extract
```

Then verify the extracted entry has no leading `\n` or trailing whitespace by checking `openlibrary/i18n/messages.pot`.

#### Short strings: prefer `{{ _() }}` over `{% trans %}`

For single words or short phrases without HTML, `{{ _('…') }}` is cleaner:

```jinja
<summary>{{ _('More') }}</summary>
<a title="{{ _('Edit this page') }}">Edit</a>
```

Reserve `{% trans %}...{% endtrans %}` for strings that need multiple lines, HTML via `|safe` variables, or pluralization with `ngettext`.

#### Pluralization

Use `{{ ngettext('singular', 'plural', count) }}` for plural strings:

```jinja
{{ ngettext('%(count)d book', '%(count)d books', books|length) }}
```

### Key differences from Templetor

| Aspect | Templetor (`.html`) | Jinja (`.jinja`) |
|---|---|---|
| Translation syntax | `$_("string")` | `{{ _('string') }}`, `{% trans %}...{% endtrans %}` |
| Formatting | Python `%` inside `$_()` | Jinja newstyle (named placeholders only, autoescaping-aware) |
| Plural | `$ungettext(s1, s2, n, key=val)` | `{{ ngettext(s1, s2, n) }}` |
| HTML in strings | `$_("...<a href>...</a>.") \| safe` | `{{ _('...<a href>...</a>.') \| safe }}` (simple) — `{% trans %}...{% endtrans %}` with `\|safe` variables (multi-line only) |
| Extraction method | `openlibrary.i18n:extract_templetor` | `jinja2.ext:babel_extract` |
| Registration | `web.template.Template.globals["_"]` | `install_gettext_callables` (auto-registers `_`, `gettext`, `ngettext`) |
