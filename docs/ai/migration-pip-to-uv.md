# Migration: pip → uv

> **Date:** June 2026
> **PR:** Moved from pip requirements files to `uv` with dependencies declared in `pyproject.toml`.

## What Changed

Open Library now uses **uv** as its sole Python package manager. We no longer use `pip` for dependency management.

### Files Removed

| File | Purpose | Replaced By |
|---|---|---|
| `requirements.txt` | Main dependencies | `[project.dependencies]` in `pyproject.toml` |
| `requirements_test.txt` | Test dependencies | `[project.optional-dependencies]` → `test` group |
| `requirements_scripts.txt` | Standalone script deps | `[project.optional-dependencies]` → `scripts` group |

### Files Added

| File | Purpose |
|---|---|
| `uv.lock` | Lockfile (auto-generated by `uv lock`) |

### Files Modified

| File | Change |
|---|---|
| `pyproject.toml` | Added `[build-system]`, `[project.dependencies]`, `[project.optional-dependencies]`, and `[tool.uv.sources]` |
| `docker/Dockerfile.olbase` | `uv pip install -r requirements.txt` → `uv sync --no-install-project` + `chown .venv` |
| `docker/Dockerfile.oldev` | `uv pip install -r requirements_test.txt` → `uv export --frozen --extra test | uv pip install -r -` |
| `scripts/solr_builder/Dockerfile.olpython` | Same pattern |
| `scripts/monitoring/Dockerfile` | Now uses `uv export --frozen --extra monitoring | uv pip install -r -` |
| .github/workflows/python_tests.yml | Removed pip fallback; uses `uv sync --extra test --no-install-project` |
| `.github/workflows/new_comment_digest.yml` | `pip install` → `uv pip install` |
| `renovate.json` | `pip_requirements` manager → `uv` manager |
| `Makefile` | Removed `test-py-uv` (now just `test-py` using `uv run --extra test`); `lint` uses `uv run --extra test ruff`; `i18n` unchanged (bare `python`) |

## Key Concepts

### Dependency Groups in pyproject.toml

Dependencies are organized into groups under `[project.optional-dependencies]`:

```toml
# Main dependencies — always installed
[project.dependencies]
fastapi = "0.136.3"
requests = "2.34.2"
# ...

# Optional groups — only installed when explicitly requested
[project.optional-dependencies]
test = ["pytest==9.0.3", "ruff==0.15.16", ...]
scripts = ["mwparserfromhell==0.7.2", ...]
monitoring = ["py-spy==0.4.2", ...]
```

### Git Dependencies via `[tool.uv.sources]`

Git-based dependencies (previously `git+https://...` in requirements.txt) are declared with `[tool.uv.sources]`:

```toml
[project.dependencies]
"web.py" = "*"
pyopds2 = "*"
"pyopds2-openlibrary" = "*"

[tool.uv.sources]
"web.py" = { git = "https://github.com/webpy/webpy.git", rev = "2598dfe693ff931a9c67196f0b904b13637ec19f" }
pyopds2 = { git = "https://github.com/ArchiveLabs/pyopds2.git", rev = "3d5b3b0f9d993b5385be013de105f98bb7d3b203" }
"pyopds2-openlibrary" = { git = "https://github.com/ArchiveLabs/pyopds2_openlibrary.git", rev = "e18e79f9a06afeaabe59d7dd8d50b1646db0646c" }
```

> **Note:** Package names with dots (`.`) or hyphens (`-`) must be quoted in TOML.

## Developer Workflow

### Before Committing

Run pre-commit on your changed files:

```bash
pre-commit run --files <file1> <file2> ...
```

### Running Tests

```bash
# Run all Python tests (uses uv automatically)
make test-py

# Run a subset
make test-py PYTEST_ARGS="openlibrary/tests/fastapi/"

# Or directly with uv
uv run --extra test pytest openlibrary/tests/fastapi/
```

### Running Lints

```bash
make lint        # ruff (via uv run)
npm run lint     # JS + CSS
```

### Installing Additional Dependencies

Instead of editing `requirements.txt`, add the dependency to the appropriate section in `pyproject.toml`:

```bash
# Add a dependency
uv add requests

# Add a dev/test dependency
uv add --group test pytest

# Run uv lock after manual edits
uv lock
```

### Running Standalone Scripts

Scripts that need the `scripts` dependency group:

```bash
uv sync --extra scripts
PYTHONPATH=. python ./scripts/providers/import_wikisource.py /olsystem/etc/openlibrary.yml
```

## Docker Build Changes

Docker builds use `uv sync` with a `chown` step to handle multi-user permissions:

```dockerfile
# Install only production dependencies
RUN uv sync --no-install-project && chown -R openlibrary:openlibrary .venv

# Install production + test dependencies
RUN uv sync --no-install-project --extra test && chown -R openlibrary:openlibrary .venv

# Install production + monitoring dependencies
RUN uv sync --no-install-project --extra monitoring && chown -R openlibrary:openlibrary .venv
```

The `--no-install-project` flag tells uv to install dependencies without building the openlibrary package itself (which runs via PYTHONPATH). The `chown` step ensures the `.venv` directory (created by `uv sync`) is accessible by the `openlibrary` user after `root` installs the packages.

### Why `.venv` Needs a Named Volume in Local Dev

In `compose.override.yaml`, each service bind-mounts the host source tree:
```yaml
- ${OL_MOUNT_DIR:-.}:/openlibrary
```
This enables live code reload — edits on the host appear instantly in the container. But it also **hides** the image's `/openlibrary/.venv` (built at `Dockerfile.olbase:80`), replacing it with whatever the host has at `./.venv`.

The host's `./.venv` either:
- Doesn't exist → no Python env → everything breaks
- Exists but was `uv sync`'d for macOS/ARM → incompatible with the Debian container

The fix is a named volume mounted **before** the bind mount:
```yaml
- ol-venv:/openlibrary/.venv
- ${OL_MOUNT_DIR:-.}:/openlibrary
```
Docker merges overlapping mounts, so the more specific path (`.venv`) stays as a persistent Docker volume with container-built deps, while the bind mount supplies everything else. This is a standard Python-in-Docker pattern: the named volume preserves the container-architected `.venv` despite the source-tree bind mount.

## CI Changes

GitHub Actions no longer has a pip fallback. The install step is now:

```yaml
- name: Install uv
  uses: astral-sh/setup-uv@v8.1.0
- name: Install dependencies
  run: |
    uv sync --extra test --no-install-project
```

(Unlike Docker, CI runs as a single user, so `.venv` permissions aren't an issue.)

## Renovate

Renovate now uses the `uv` manager instead of `pip_requirements`. It will read the `uv.lock` file for dependency updates. The `uv.lock` file is the single source of truth for all resolved versions.

## FAQ

### Can I still use pip locally?

Yes, but you shouldn't need to. The `uv.lock` file pins exact versions for reproducibility. If you use `pip install -r <(uv export)`, it works, but we strongly recommend using `uv` directly.

### What about the `vendor/infogami` directory?

Infogami is a vendored submodule that's symlinked into the Python path. It's not installed as a pip/uv package. This behavior is unchanged.

### Why no `setup.py` replacement?

The `setup.py` at the project root is only used by `scripts/solr_builder/build-cython.sh` to compile Cython extensions for the Solr builder. We kept `setup.py` for this purpose and added explicit `uv pip install setuptools Cython` to the build script. The normal `uv sync` path uses hatchling as a lightweight build backend.

**What is hatchling?** It's a PEP 517 build backend declared in `pyproject.toml`:
```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```
When `uv sync` installs the root project (without `--no-install-project`), it calls hatchling to build a wheel from `pyproject.toml` metadata — no compilation, just packaging. It replaced setuptools because it's zero-config and has no runtime dependencies beyond Python itself. Alternatives like `setuptools`, `flit_core`, or `pdm-backend` would also work, but hatchling is the simplest option for a pure-Python project that doesn't need complex build steps.

### How do I update a dependency version?

1. Edit the version in `pyproject.toml`
2. Run `uv lock` to update the lockfile
3. Commit both `pyproject.toml` and `uv.lock`
